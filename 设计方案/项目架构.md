项目架构（执行版）20260415

定位：手机端竖屏、轻量级、无后端网页游戏（Godot 导出 HTML）。
目标：先把框架和接口跑通，再逐步替换占位逻辑。
当前范围：不实现复杂卡牌代码与敌人意图状态机，仅预留接口与占位实现。

1. 架构总原则

1.1 模块单向依赖，避免循环引用。
1.2 模块之间通过统一通过信号系统通信，不直接改写对方内部状态。
1.3 先保证流程闭环，再迭代数值与复杂机制。
1.4 所有占位逻辑必须可替换，不允许把占位写死在主流程里。

2. 模块与职责

2.1 地图控制器
- 负责节点序列生成、节点切换、层内推进。
- 负责把当前节点路由到战斗、事件、商店、休息处场景。
- 战斗结束后接收结果并推进到下一个节点。

2.2 战斗控制器
- 负责固定回合阶段推进（进入战斗、回合开始、意图展示、道具窗口、摇老虎机、技能结算、敌人行动）。
- 负责战场状态维护（玩家、敌人、格挡、状态、回合计数）。
- 负责调用老虎机模块并消费抽取结果。
- 负责输出战斗结果给地图与奖励模块。
- 敌人行为先用固定模板占位，不做意图状态机。

2.3 老虎机模块
- 负责三轮独立随机池的数据与抽取。
- 负责回合抽取结果对象生成。
- 负责战后技能加入某轮随机池的写入接口。
- 轮子数量参数化，当前默认 3 轮，预留未来扩展能力。

2.4 奖励模块
- 负责战后奖励展示与选择。
- 负责奖励结果转为标准变更请求（如：加入随机池、获得道具、获得饰品）。
- 不直接操作战斗内部数据，只通过接口提交变更。

2.5 全局效果模块
- 负责跨战斗持续生效的效果（如饰品、局外解锁加成）。
- 负责在战斗开始时注入效果，在战斗结束时回收和同步状态。
- 负责统一效果生命周期管理，避免分散在各处。

2.6 一次性道具模块
- 负责道具库存、使用条件与消耗。
- 仅在摇老虎机前窗口可触发。
- 通过接口影响本回合（如重摇、替换、加成），不直接修改战斗阶段机。

2.7 UI 与表现模块
- 负责竖屏 HUD、奖励界面、节点界面、占位事件界面、占位商店界面。
- 负责战斗可视反馈（抽取结果、伤害数字、状态变化）。
- 负责输入映射（点击、触摸）与主流程交互同步。

2.8 数据配置模块
- 负责基础数据定义与加载（技能模板、敌人模板、饰品模板、道具模板、地图模板）。
- 负责调参入口与默认值回退策略。
- 占位阶段只需要最小数据集支持流程闭环。

2.9 存储模块
- 负责单局续玩存档（局内进度）。
- 负责局外解锁存档（长期进度）。
- 负责存档版本字段和兼容处理，避免迭代后坏档。

3. 任务拆分（可执行）

3.1 A-基础骨架任务
- 建立主流程入口和全局上下文。
- 建立事件通信机制和调试日志。
- 建立最小配置加载链路。
- 完成标准：可启动并进入地图到战斗到结算空跑流程。

3.2 B-战斗主循环任务
- 完成固定回合阶段机。
- 接入玩家和敌人基础状态。
- 接入敌人固定模板行为占位。
- 接入技能最小占位（攻击、防御、位移）。
- 完成标准：可进行多回合并稳定结束战斗。

3.3 C-老虎机与奖励任务
- 完成三轮独立随机池与抽取。
- 完成抽取结果展示与结算调用。
- 完成战后奖励选择与入池。
- 完成标准：战斗结果可改变下一场战斗的随机池构成。

3.4 D-地图与节点流转任务
- 完成层内节点序列与跳转。
- 完成战斗节点和非战斗节点占位接入。
- 完成标准：可在一层内连续推进多个节点。

3.5 E-存档与恢复任务
- 完成单局续玩存档。
- 完成局外解锁存档。
- 完成标准：中断后可恢复到正确节点并继续。

3.6 F-UI竖屏与部署任务
- 完成竖屏关键界面可用性。
- 完成 Godot HTML 导出流程。
- 完成 Cloudflare Pages 部署流程。
- 完成标准：移动端浏览器可运行基础完整流程。

4. 依赖关系

4.1 战斗阶段机先于技能占位与敌人占位。
4.2 全局状态模型先于存档系统。
4.3 节点切换协议先于事件与商店扩展。
4.4 老虎机抽取结果接口先于奖励回写接口。
4.5 UI 可以并行开发，但必须依赖稳定的接口对象。

5. 完成定义（DoD）

5.1 每个模块必须有清晰输入、输出、边界。
5.2 每个任务必须能被独立验收（流程验收 + 接口验收）。
5.3 不允许把复杂机制提前混入主流程。
5.4 占位逻辑可随时替换，不影响其他模块。
5.5 主流程在手机竖屏 HTML 目标下可连续运行。

6. 当前明确不做

6.1 复杂卡牌脚本系统。
6.2 敌人意图状态机。
6.3 高级 Combo 递归规则与精细数值平衡。
6.4 开发优先级、占位替换触发条件、命名规范细则（由制作人手动决定）。
