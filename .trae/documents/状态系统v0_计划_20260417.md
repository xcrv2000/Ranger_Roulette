# 状态系统 v0 计划（战斗内 Buff/Debuff）

## Summary

在现有战斗框架上新增“状态（Status）”底层系统，用于承载玩家/敌人的 buff、debuff、局内临时状态：每个状态包含名字、层数（stacks）、效果逻辑；状态仅在单场战斗内存在，战斗结束清空。

本计划优先把 **数据结构 + 触发接口（Hook）+ 可调度管线**搭起来；具体某个 buff 的效果（例如易伤/虚弱/中毒）后续按需求逐个实现。

## 底层逻辑设计（你要的“怎么做”）

### 1) 角色身上的状态 = StatusContainer（按 id 管理 StatusInstance）

每个战斗实体（玩家、每个敌人）拥有一个 `StatusContainer`：

- `Dictionary _by_id`：key = status\_id，value = StatusInstance
- 对外 API：
  - `add(status_id: String, stacks: int = 1, source: Dictionary = {})`
  - `set(status_id, stacks)` / `remove(status_id)` / `get_stacks(status_id)`
  - `get_all(): Array[StatusInstance]`（用于 UI 展示）
- 规则（v0）：
  - 只有 stacks（层数）
  - 默认叠加：`stacks += delta`，若 `stacks <= 0` 自动移除
  - 战斗开始创建，战斗结束销毁（不写入 RunContext）

### 2) StatusInstance = “数据 + 行为脚本”

StatusInstance 包含：

- `id: String`
- `name: String`
- `description: String`
- `stacks: int`
- `script: Script`（或 effect 对象）

“行为脚本”的职责：提供一组可选 Hook 方法（类似杀戮尖塔的触发点），在不同阶段被调用。

### 3) Hook 设计：事件驱动 + 可修改的上下文字典（避免强耦合）

核心目标：让状态效果不直接依赖 BattleController 内部变量名，同时又能读写必要信息。

统一采用：

- `event_name: String`
- `ctx: Dictionary`（上下文，包含 actor/target/skill\_id/intent 等）
- `payload: Dictionary`（可变载荷，用于“修改数值/记录效果”）

建议的最小 Hook 集（先实现调度接口，具体效果后续再写）：

- **回合阶段**
  - `on_turn_start(ctx, payload)`
  - `on_turn_end(ctx, payload)`
- **技能结算**
  - `before_skill(ctx, payload)`：可修改 `payload.effect`（damage/block/move）
  - `after_skill(ctx, payload)`：仅记录/触发副作用
- **伤害前后**
  - `before_deal_damage(ctx, payload)`：修改 outgoing damage
  - `before_take_damage(ctx, payload)`：修改 incoming damage
  - `after_damage(ctx, payload)`：统一后置
- **移动**
  - `on_move(ctx, payload)`：例如“每移动 1 格触发伤害”

调度顺序（确定性）：

-  采用 **按状态添加顺序**&#x20;

### 4) BattleController 变成“管线编排者”

BattleController 的数值结算不再直接写死：

- 生成 `ctx/payload`
- 让 `StatusContainer.dispatch(event_name, ctx, payload)` 先/后处理
- 再把最终值应用到 hp/block/move

示例（概念）：

- 玩家释放技能：
  1. 解析卡牌 `effect = {damage, block, move}`
  2. `before_skill`（玩家状态）→ 可改 effect
  3. 应用 block/move
  4. 造成伤害前：
     - `before_deal_damage`（玩家状态）
     - `before_take_damage`（敌人状态）
  5. 应用伤害
  6. `after_skill` / `after_damage`

这样任何 buff/debuff 都只需要实现自己的 Hook，不需要改 BattleController 主体。

### 5) 数据化（与 cards/enemies 同风格）

新增 `data/statuses.json`：

- 每个状态：`id/name/icon/anim_module/script`
- anim\_module 先留空，和卡牌/敌人一致

后续新增状态，只需：

1. JSON 添加定义
2. scripts/statuses/ 添加脚本实现 Hook

## Assumptions & Decisions

- 状态仅在单场战斗内生效，战斗结束清空（不写入 RunContext）。
- 状态只包含 stacks（层数），不引入 duration/data（后续需要再扩展）。
- v0 先把 Hook 接口与调度管线做完整；具体状态效果按后续卡牌/敌人需求逐个实现。
- 为避免 warning-as-error，所有 `:=` 推断 Variant 的地方会改为显式类型标注。
