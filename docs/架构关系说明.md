# 架构关系说明

## 1. 本次落地目标

本次仅完成框架级搭建：

- 建立可维护的目录结构。
- 在 Godot 中创建空节点并完成重命名。
- 将主场景接入项目启动配置。
- 明确模块之间的职责和依赖方向。

当前阶段不实现复杂卡牌逻辑和敌人意图状态机，相关能力用占位节点承载。

## 2. 目录结构

```text
scenes/
  core/
    main.tscn
  map/
    map_root.tscn
  battle/
    battle_root.tscn
    slot_root.tscn
  ui/
    ui_root.tscn
scripts/
  core/
  map/
  battle/
  systems/
  ui/
data/
  config/
docs/
  架构关系说明.md
```

说明：

- scenes 放场景骨架和节点组织。
- scripts 预留给对应模块脚本实现。
- data/config 预留数据配置。
- docs 放架构与协作说明。

## 3. 主场景节点关系

主场景：`res://scenes/core/main.tscn`

根节点：Main

一级子节点：

- GameFlow
- MapController
- BattleController
- SlotController
- RewardController
- GlobalEffects
- ConsumableController
- UIController
- DataConfig
- SaveSystem

关系说明：

- Main 负责组装和生命周期管理。
- GameFlow 负责高层流程推进。
- MapController 负责地图节点流转。
- BattleController 负责战斗阶段机和战场状态。
- SlotController 负责老虎机抽取与随机池。
- RewardController 负责战后奖励回写。
- GlobalEffects 负责跨战斗常驻效果。
- ConsumableController 负责一次性道具时机与消耗。
- UIController 负责界面层路由与展示。
- DataConfig 负责配置装载与分发。
- SaveSystem 负责局内/局外存档。

## 5. 依赖方向

依赖采用单向流：

- DataConfig -> MapController / BattleController / SlotController / RewardController
- GameFlow -> MapController -> BattleController -> RewardController -> MapController
- BattleController -> SlotController
- GlobalEffects -> BattleController
- ConsumableController -> BattleController
- SaveSystem -> GameFlow / MapController / BattleController（序列化与恢复）
- UIController 订阅各模块状态，不反向写入核心流程
